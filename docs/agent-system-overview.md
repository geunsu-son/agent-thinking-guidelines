# 에이전트 시스템 개요

> 이 문서는 repo에 구현된 지침 세트가 **실제로 어떻게 동작하는지**를 설명한다.
> 세부 규약은 [`agent-thinking-guidelines.md`](agent-thinking-guidelines.md)(기본 지침)과 [`multi-agent-orchestration.md`](multi-agent-orchestration.md)(멀티에이전트 확장)을 참조한다.

---

## 1. 한 줄 요약

**메인 에이전트**가 Worker·라우터·상태 관리자 역할을 맡고, **orchestrator**는 계획만, **reviewer**는 검증만 담당하는 **역할 분리 + 컨텍스트 격리** 구조다. Rules/Skills가 모든 작업의 바닥을 깔고, 서브에이전트는 필요할 때만 올라탄다.

---

## 2. 전체 구조

```
[사용자]
   │  작업 의뢰 / 승인 / 에스컬레이션 응답
   ▼
[메인 에이전트] ─── Worker 역할 겸임, 실제 라우팅·상태 추적
   │
   ├─▶ [orchestrator 서브에이전트] ─── 분해·위험등급·라우팅 계획만 반환 (선택)
   │
   ├─▶ [Worker] ─── 산출물 작성 (기본적으로 메인 에이전트가 수행)
   │        │ 산출물 + 자기신고(가정 목록, 미해결 사항)
   │        ▼
   └─▶ [reviewer 서브에이전트] ─── 체크리스트 기반 판정 (통과 / 반려+수정지시)
            │
            ├─ 통과 → 다음 단계 or 위험 등급별 사용자 확인
            └─ 반려 → 수정 지시서와 함께 Worker로 반환
```

### 구현상 핵심

| 역할 | 담당 주체 | 하는 일 | 하지 않는 일 |
|---|---|---|---|
| **오케스트레이터** | orchestrator 서브에이전트 | 작업 분해, 위험 등급, 라우팅 **계획** 반환 | 산출물 작성, 서브에이전트 직접 호출 |
| **Worker** | 메인 에이전트 (기본) | 산출물 작성, 자기신고 | — |
| **승인자** | reviewer 서브에이전트 | 체크리스트 판정, 수정 지시 | 산출물 직접 수정 |
| **실행·라우팅** | 메인 에이전트 | Worker 진행, reviewer 호출, 상태표 갱신, 에스컬레이션 | — |

서브에이전트가 다른 서브에이전트를 중첩 호출하는 것은 플랫폼·버전에 따라 불안정하거나 불가능하므로, **실제 라우팅 주체는 항상 메인 에이전트**로 고정한다.

## 3. 지침 계층

repo는 Cursor·Claude Code 두 버전으로 매핑되어 있으며, 역할 구조는 동일하다.

| 계층 | Cursor | Claude Code | 적용 방식 |
|---|---|---|---|
| **기본 원칙** | `core-principles.mdc` | `CLAUDE.md` | 항상 적용 |
| **작업 규율** | `worker-conduct.mdc` | `CLAUDE.md` | 항상 적용 |
| **상황별 프로토콜** | `skills/analysis-protocol`, `skills/design-protocol` | 동일 경로 | 작업 유형에 따라 트리거 |
| **플래너** | `agents/orchestrator.md` | `agents/orchestrator.md` | 명시 호출 (`/orchestrator`) |
| **승인자** | `agents/reviewer.md` | `agents/reviewer.md` | 명시 호출 (`/reviewer`) |
| **세션 간 기억** | `.cursor/memory/` | `.claude/memory/` | 수동 참조 |

일상 작업에서는 **메인 에이전트 + 기본 원칙·작업 규율**만으로도 동작한다. reviewer는 중요 산출물에, orchestrator는 대규모·다단계 작업에 추가한다.

## 4. 동작 프로세스

### Phase 0 — 착수

1. 사용자가 `[목적] [작업] [입력] [제약]` 형태로 요청한다.
2. (선택) orchestrator에 작업 분해·위험 등급·라우팅 계획을 요청한다.
3. orchestrator가 반환하는 것:
   - 작업 목록 (T-001, T-002, …) + 의존 순서
   - 각 작업의 위험 등급 (하/중/상)
   - 완료 기준이 포함된 작업 지시서 초안
   - 초기 상태표 초안, 루프·종료 규칙
4. 메인 에이전트가 분해 결과를 **사용자에게 먼저 보고**하고 승인을 받는다. (첫 체크포인트)
5. `loop-status.md`를 생성한다.

### Phase 1 — Worker (산출물 작성)

메인 에이전트가 Worker 역할을 수행한다.

1. 작업 지시서의 **완료 기준**을 시작 전에 읽는다.
2. 모호함 처리 모드에 따라 진행한다.
   - **질문 우선**: 핵심 모호점은 질문
   - **가정 후 명시**: 합리적 기본값 + 결과에 가정 명시
3. 상황에 맞는 Skill을 적용한다. (분석 → 샘플 검증 후 전체 확장, 설계 → 구조 먼저 합의)
4. 산출물 완성 후 **자체 검토** (6항목 체크리스트, `agent-thinking-guidelines.md` 5장).
5. 아래 형식의 **산출물 제출서**로 제출한다.

```
[작업ID] T-001
[산출물] (본문 또는 파일 경로)
[가정 목록] ...
[미해결 사항] ...
[자체 검토 결과] ...
```

의존 관계 없는 독립 작업만 **병렬 서브에이전트**에 위임할 수 있다. 같은 파일 동시 수정, 이전 결과에 의존하는 작업은 병렬 불가.

### Phase 2 — Reviewer (품질 게이트)

1. 메인 에이전트가 reviewer 서브에이전트를 호출한다.
2. reviewer 입력: **요구사항 원문 + 산출물 + 체크리스트**. 작성자의 추론 과정은 받지 않는다. (맹점 공유 방지)
3. reviewer는 `readonly`로 실행되므로 **판정만** 하고 직접 수정하지 않는다.
4. 체크리스트를 **항목별** 통과/실패 판정하고, 산출물에서 근거를 인용한다.
5. 수치 산출물은 최소 1개 **역산 검증** 기록을 남긴다.
6. 아래 형식의 **판정서**를 반환한다.

```
[판정] 통과 / 반려 / 판정 불가
[판정표] 항목별 통과/실패 + 근거 인용
[수정 지시] (반려 시) 어디가 / 왜 / 어떻게
```

### Phase 3 — 루프 관리

메인 에이전트가 매 판정 후 `loop-status.md`를 **즉시 갱신**한다.

| 판정 | 다음 동작 |
|---|---|
| **통과** | 다음 작업 진행. 위험 등급에 따라 사용자 확인 |
| **반려 (1~2회)** | 수정 지시와 함께 Worker 재작업 (지적 항목만 수정) |
| **같은 사유 2연속 반려** | 즉시 사용자에게 에스컬레이션 |
| **반려 3회** | 즉시 에스컬레이션 |
| **Worker "해소 불가"** | 즉시 에스컬레이션 |
| **reviewer "판정 불가"** | 즉시 에스컬레이션 |

에스컬레이션은 실패가 아니라 **설계된 정상 출구**다. 시도 이력 + 패턴 진단 + 선택지(요구사항 완화 / 자료 추가 / 사람이 직접 처리)를 사용자에게 제시한다.

### Phase 4 — 위험 등급별 최종 승인

| 등급 | 기준 | 최종 승인 |
|---|---|---|
| **하** | 되돌리기 쉬움 (초안, 내부 메모) | reviewer 전결 |
| **중** | 재작업 비용 큼 | reviewer 통과 → 사용자 요약 보고 (이의 없으면 진행) |
| **상** | 되돌리기 어렵거나 외부 전달 (DB 변경, 대외 공지) | reviewer 통과 후에도 **사용자 명시 승인 필수** |

등급이 애매하면 한 단계 높은 쪽으로 처리한다.

## 5. 운용 모드

도입은 단계적으로 권장한다 (`multi-agent-orchestration.md` 9장).

```
1단계: 메인 에이전트(Worker) + reviewer만
       → 판정 품질·반려 패턴 관찰
2단계: 체크리스트 보정 (오판 사례 반영)
3단계: orchestrator 플래너 추가
       → 대규모·다단계 작업에만 사용
```

| 모드 | 구성 | 적합한 작업 |
|---|---|---|
| **단순** | 기본 원칙 + 메인 에이전트 | 일상 코드 수정, 소규모 분석 |
| **검증** | + reviewer | 수치 보고, 되돌리기 어려운 변경 |
| **풀 루프** | + orchestrator + state | 대규모 리팩터링, 다단계 설계 |

## 6. 통신 규약

모든 역할 간 통신은 **정해진 메시지 형식**을 따른다. 자유 서술 전달은 금지한다. 상세 형식은 `multi-agent-orchestration.md` 2장.

| 메시지 | 방향 | 핵심 필드 |
|---|---|---|
| **작업 지시서** | orchestrator → Worker | 목적, 작업, 입력, 제약, 완료 기준, 모호함 처리 모드 |
| **산출물 제출서** | Worker → reviewer | 산출물, 가정 목록, 미해결 사항, 자체 검토 |
| **판정서** | reviewer → 메인 | 판정, 체크리스트 판정표, 수정 지시 |

가정을 **정직하게 신고**하면 반려 사유가 되지 않는다. 숨긴 가정이 발각되면 **신뢰 위반**으로 기록한다.

## 7. 세션 간 연속성

### Memory (`.cursor/memory/` / `.claude/memory/`)

- 프로젝트별 교훈을 파일로 축적한다. (한 교훈 = 한 파일)
- 세션 시작 시 각 파일의 `요약:` 한 줄만 스캔하고, 관련 주제만 본문을 연다.
- `docs/` 지침과 충돌하면 지침이 우선한다.

### State (`loop-status.md`)

- 3-에이전트 루프 실행 시 작업 상태·반려 횟수를 파일로 영속화한다.
- **매 판정**(통과/반려/에스컬레이션) 후 즉시 갱신한다.
- 세션 재개·컨텍스트 압축 후에는 컨텍스트보다 이 파일을 먼저 읽는다.

상태 값: `대기` / `진행` / `검토중` / `반려(n회)` / `통과` / `에스컬레이션`

## 9. 알려진 한계

- 지침은 행동 패턴을 교정하지만 **모델의 판단 능력 자체**를 올리지는 못한다.
- reviewer와 Worker가 같은 모델이면 맹점을 공유할 수 있다. 통과된 산출물도 주기적으로 사람이 샘플 검수해야 한다.
- 되돌리기 어려운 작업(DB 변경, 외부 전달물)은 지침·reviewer와 무관하게 **사람이 최종 확인**한다.
