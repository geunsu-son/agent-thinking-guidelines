# 설계 작업 프로토콜

## 착수 전
- 해결하려는 문제를 한 문장으로 정의한다.
- 제약 조건을 목록화한다: 기술 / 일정 / 정책 / 기존 시스템 호환.
- 기존 코드·문서에서 재사용할 것과 새로 만들 것을 구분한다.

## 설계 중
- **구조(목차·컴포넌트 구성·변경 대상 목록)를 먼저 제시하고 사용자 확인을 받은 뒤 상세를 채운다.** 대규모 산출물을 확인 없이 한 번에 완성하지 않는다.
- 결정과 미결정을 구분해 문서화한다. 미결정은 "열린 결정사항" 섹션에 옵션 + 트레이드오프로 남긴다.
- 주요 결정마다 선택하지 않은 대안과 이유를 1줄이라도 기록한다.
- 엣지 케이스를 설계 단계에서 나열한다: 빈 값, 중복, 동시 발생, 권한 없음, 중간 이탈, 롤백.
- 확장은 1단계까지만 고려한다. 과잉 설계 금지, 막다른 구조도 금지.

## 코드 설계·수정 시 추가
- 기존 코드 스타일·패턴을 먼저 파악하고 따른다. 임의로 새 패턴 도입하지 않는다.
- 수정 범위를 최소화한다. 지시받지 않은 리팩터링을 끼워넣지 않는다.
- 변경이 영향을 주는 다른 파일·호출부를 확인하고 보고한다.

## 판단 예시
- 나쁨: 요청받은 기능을 만들며 주변 코드도 "개선"해서 함께 수정
- 좋음: 요청 범위만 수정, 발견한 주변 문제는 별도로 보고만
- 나쁨: 43개 항목 전체를 한 번에 분류해 완성본 제출
- 좋음: 분류 기준 제시 → 합의 → 5개 샘플 적용 → 확인 → 전체 확장
